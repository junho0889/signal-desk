## Flutter Layer Plan (v0)
### 1) Presentation Layer
- feature modules:
  - `home`
  - `ranking`
  - `keyword_detail`
  - `watchlist`
  - `alerts`
- each feature owns:
  - screen widgets
  - view state model
  - user interaction intents

### 2) State Layer
- recommended baseline: `Riverpod` providers per feature (state + async query state)
- shared state objects:
  - filter state (`period`, `market`, `sector`)
  - pagination cursors (`keywords`, `alerts`)
  - watchlist mutation in-flight state
- state shape requirements:
  - explicit `loading`, `data`, `error`, `empty` branches
  - nullable fields preserved as nullable in model types

### 3) Data Layer
- repository interfaces by contract domain:
  - `DashboardRepository`
  - `KeywordRepository`
  - `WatchlistRepository`
  - `AlertRepository`
- repository responsibilities:
  - call API endpoints
  - decode DTOs
  - map DTO -> UI-safe models
  - enforce enum/value guards (`risk_flags`, severity, period, market)

### 4) Network Layer
- HTTP client with request/response logging and timeout policy
- base URL points to `/v1`
- common error parser for `error.code`, `error.message`, `error.details`, `request_id`
- retry policy only for transient GET failures (idempotent reads)

## Screen To Endpoint Mapping
| Screen | Endpoint | Required Params | Key Bindings |
|---|---|---|---|
| Home | `GET /dashboard` | none | `top_keywords[]` -> cards, `hot_sectors[]` -> movers, `risk_alerts[]` -> summary |
| Keyword Ranking | `GET /keywords` | `period`; optional `market`,`sector`,`limit`,`cursor` | list rows use `rank_position`,`score`,`delta_1d`,`confidence`,`is_alert_eligible`,`reason_tags`,`risk_flags` |
| Keyword Detail | `GET /keywords/{keyword_id}` | path `keyword_id`; optional `period`,`points` | header from `score_summary`, chart from `timeseries[]`, evidence from `related_news[]`,`related_stocks[]`,`related_sectors[]` |
| Watchlist | `GET /watchlist` + `POST /watchlist` | mutation body: `op`,`target_type`,`target_id` | `keywords[]` and `stocks[]` tabs; add/remove interactions from detail/list |
| Alerts | `GET /alerts` | optional `limit`,`cursor`,`severity` | list rows from `items[]`; detail deep-link via `keyword_id` or `target_id` |

## Loading/Error/Empty Strategy
- global rule: never hide stale data silently; show last render + explicit refresh state where possible
- loading:
  - full-screen skeleton on first load
  - inline loading indicators for pagination and watchlist mutations
- error:
  - map API `error.code` to user-safe message buckets (`network`, `invalid_argument`, `server`)
  - expose retry actions for read flows
- empty:
  - ranking empty: show filter reset prompt
  - watchlist empty: route to Ranking/Detail add flow
  - alerts empty: show "no recent triggers" neutral state
- nullable rendering:
  - numeric null -> `-`
  - optional arrays null/empty -> empty chips/list placeholder
  - `is_alert_eligible` nullable in watchlist keyword rows -> show `unknown` badge when null

## Push Notification UX Plan
- push payload opens Alerts list by default
- deep-link rules:
  - if payload includes `keyword_id`: open Keyword Detail directly
  - else open Alerts row context then allow detail navigation
- app receives push as review prompt only; no auto-action or trade instruction text
