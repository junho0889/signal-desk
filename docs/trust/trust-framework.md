# Trust Framework (v1)

## Purpose
Define an explainable and auditable trust layer for SignalDesk that scores information quality without collapsing uncertainty into a binary verdict.

## Scope And Non-Goals
- Scope:
  - score trust dimensions for each keyword window (`keyword_id`, `as_of_ts`)
  - expose contradiction and misinformation-risk signals for storage, model, and app use
  - provide warning candidates that the app can render to users
- Non-goals:
  - no end-to-end ML classifier implementation in this task
  - no source censorship logic (we surface risk, we do not hide evidence by default)
  - no hard freeze of physical table names before storage lane (`BE-004`) finalizes schema

## Inputs
- raw payload lineage and source metadata from collector/storage lanes
- normalized events and dedup clusters from data layer
- `source_coverage_ratio`, `freshness_minutes`, `quality_flag`, and `risk_flag` signals from data layer
- entity mapping confidence (`keyword -> symbol/sector`) from normalization outputs

## Trust Principles
- Keep trust inspectable through per-dimension scores and evidence counts.
- Preserve contradictory evidence and make disagreement explicit.
- Separate low coverage from low credibility.
- Version trust outputs and reasons; do not silently overwrite prior assessments.

## Trust Dimensions
Each dimension is scored `0-100` per (`keyword_id`, `as_of_ts`), where higher is better for trust except contradiction and syndication dimensions (explicitly marked as risk dimensions).

| dimension_id | Type | What It Captures | Primary Inputs | Suggested v1 Interpretation |
|---|---|---|---|---|
| `source_reliability_score` | quality | historical credibility of contributing sources | source registry tier, correction history, ingestion health | `80+` strong, `50-79` mixed, `<50` weak |
| `freshness_score` | quality | timeliness relative to source cadence | `freshness_minutes`, source target freshness | `80+` fresh, `50-79` aging, `<50` stale |
| `source_diversity_score` | quality | independence and breadth of evidence | unique source count, category mix, concentration ratio | `80+` broad, `50-79` moderate, `<50` narrow |
| `entity_mapping_confidence_score` | quality | confidence in keyword-symbol/sector linkage | mapping confidence, alias stability | `80+` stable, `60-79` usable, `<60` low confidence |
| `contradiction_risk_score` | risk | unresolved disagreement in core claims | contradiction links, claim polarity mismatch, source independence | `<30` low risk, `30-59` monitor, `60+` active contradiction |
| `syndication_intensity_score` | risk | amplification from near-duplicate coverage | dedup ratio, duplicate cluster size, repeated headline patterns | `<30` low, `30-59` moderate, `60+` high syndication |

## Core Output Contract (Logical)
These fields are stable for downstream lanes even while storage physical schema is still owned by `BE-004`.

| field | Type | Notes |
|---|---|---|
| `trust_assessment_version` | string | start with `v1` |
| `keyword_id` | string | canonical keyword id |
| `as_of_ts` | timestamptz | window timestamp in UTC |
| `trust_score` | number (`0-100`) | weighted aggregate for sorting/filtering, not a binary pass/fail |
| `coverage_score` | number (`0-100`) | evidence sufficiency and source coverage signal |
| `dimension_scores` | object | per-dimension scores listed above |
| `dimension_reasons` | array | machine-readable reason codes per dimension |
| `contradiction_state` | enum | `none`, `emerging`, `active`, `resolved` |
| `misinformation_risk_score` | number (`0-100`) | aggregated risk signal from contradiction + source patterns |
| `misinformation_risk_level` | enum | `low`, `medium`, `high` |
| `quality_flags` | array | includes data-layer compatible flags: `stale_source`, `mapping_low_confidence`, `low_source_diversity`, `deduped` |
| `risk_flags` | array | includes data-layer baseline flags such as `data_freshness_degraded`, `event_coverage_partial`, `mapping_unstable`, `thin_cohort` |
| `warning_candidates` | array | app-facing warning ids generated by rules below |
| `evidence_summary` | object | counts and provenance summary used for app explainability |

## Contradiction Handling Policy
### Detection
- Build contradiction groups on normalized claim facets (`entity`, `claim_topic`, `time_bucket`).
- Mark contradiction when independent sources support materially opposing claim polarity inside the same analysis window.

### State Machine
- `none`: no meaningful contradictory evidence.
- `emerging`: first opposing evidence appears but confidence/volume is limited.
- `active`: opposing evidence from independent sources crosses review threshold.
- `resolved`: contradiction previously active but newer higher-reliability evidence converges.

### Output Behavior
- Never drop contradictory records from evidence timelines.
- Increase `contradiction_risk_score` as contradictions move from `emerging` to `active`.
- Apply trust impact as a graded penalty, not a hard invalidation.
- Attach `dimension_reasons` entries such as:
  - `contradiction_multi_source_active`
  - `contradiction_low_independence`
  - `contradiction_resolved_recently`

## Misinformation-Risk Policy
### Indicators (Heuristic v1)
- high contradiction risk with low source reliability
- high syndication intensity with low source diversity
- repeated claim propagation without primary-source support
- unstable or low-confidence entity mapping on high-impact claims
- stale data conditions combined with aggressive claim spread

### Required Outputs
- `misinformation_risk_score` (`0-100`)
- `misinformation_risk_level` (`low|medium|high`)
- `misinformation_reasons[]` reason codes, for example:
  - `misinfo_contradiction_high`
  - `misinfo_syndication_amplified`
  - `misinfo_primary_source_missing`
  - `misinfo_mapping_unstable`
  - `misinfo_stale_but_spreading`
- `misinformation_review_flag`:
  - `false` for `low`
  - `true` for `medium` and `high`

## App-Facing Warning Candidates
These warning ids are stable candidates for design/app integration and map directly from trust outputs.

| warning_id | Trigger (logical) | Severity | Suggested App Treatment |
|---|---|---|---|
| `warning_active_contradiction` | `contradiction_state=active` or `contradiction_risk_score >= 60` | high | show prominent caution label near trust badge and evidence section |
| `warning_misinfo_high_risk` | `misinformation_risk_level=high` | high | show caution card and suppress push-eligible confidence messaging |
| `warning_stale_source_data` | freshness risk flag or `freshness_score < 50` | medium | show stale timestamp warning and downgrade certainty language |
| `warning_low_source_diversity` | `source_diversity_score < 50` | medium | show "limited source variety" notice |
| `warning_mapping_low_confidence` | `entity_mapping_confidence_score < 60` or mapping risk flag | medium | warn that symbol/sector linkage may be unstable |
| `warning_event_coverage_partial` | `risk_flag=event_coverage_partial` | medium | warn that disclosure/event coverage is incomplete |
| `warning_syndication_spike` | `syndication_intensity_score >= 60` | low | indicate amplified duplicate reporting pattern |

## Storage-Facing Expectations (For BE-004 Integration)
- Persist trust outputs as versioned assessments keyed by (`keyword_id`, `as_of_ts`, `trust_assessment_version`).
- Persist dimension-level scores and reason codes separately so audits can inspect why a score changed.
- Persist contradiction links and state transitions with source references.
- Persist misinformation-risk outputs with reason codes and generation timestamp.
- Ensure lineage path can trace `trust_assessment -> normalized evidence -> raw payload hash`.

## Stable Outputs For Downstream Lanes
- Stable for design lane:
  - warning ids in `App-Facing Warning Candidates`
  - severity mapping (`low|medium|high`)
- Stable for app lane:
  - `trust_score`, `coverage_score`, `contradiction_state`, `misinformation_risk_level`, `warning_candidates`, `evidence_summary`
- Stable for model lane:
  - all `dimension_scores`
  - `misinformation_risk_score`
  - contradiction/syndication states as model features
